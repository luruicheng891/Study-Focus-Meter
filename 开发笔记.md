## 二、硬件抽象层笔记

### 2.1 时钟树配置

```
HSE = 25MHz (外部高速晶振，假设板载为 25MHz)

PLL1 配置:
  PLLM = 10  → VCO 输入 = 25MHz / 10 = 2.5MHz
  PLLN = 220 → VCO 输出 = 2.5MHz × 220 = 550MHz
  PLLP = 1   → SYSCLK = 550MHz / 1 = 550MHz
  PLLQ = 2   → PLL1Q = 275MHz
  PLLR = 2   → PLL1R = 275MHz

总线时钟:
  SYSCLK   = 550 MHz
  AHB/HCLK = 550 / 2 = 275 MHz
  APB1     = 275 / 2 = 137.5 MHz
  APB2     = 275 / 2 = 137.5 MHz
  APB3     = 275 / 2 = 137.5 MHz
  APB4     = 275 / 2 = 137.5 MHz

电压调节: VOS0 (最高性能模式)
Flash 等待周期: FLASH_LATENCY_3
```

**注意事项:**
- 使用 `PWR_LDO_SUPPLY` 供电模式
- 必须等待 `PWR_FLAG_VOSRDY` 后才能配置 PLL
- ICache 和 DCache 均已开启 (`SCB_EnableICache()` / `SCB_EnableDCache()`)

### 2.2 MPU 配置

| 区域 | 基地址 | 大小 | 属性 |
|------|--------|------|------|
| Region 0 | 0x24000000 (AXI SRAM) | 512KB | Full Access, Bufferable, Cacheable, Shareable |

> 用途: 摄像头 DMA 帧缓冲区放置在 AXI SRAM，配置为可缓存+可共享，确保 DMA 与 CPU 数据一致性。

## 三、驱动与中间件笔记

### 3.1 OV2640 摄像头驱动

- 像素格式: RGB565
- 采集分辨率: 400×300 (OV2640 输出)
- 显示分辨率: 240×240 (DCMI 硬件裁剪后)
- DMA 缓冲区: 位于 AXI SRAM (0x24000000)
- DMA 缓冲区大小: `240 × 240 × 2 / 4 = 28800` (32 位字)
- 传输模式: DMA 连续模式 (Continuous)

**帧处理流程:**
1. DCMI 中断回调设置 `DCMI_FrameState = 1`
2. Camera_task 检测到标志后:
   - 调用 `SCB_InvalidateDCache_by_Addr()` 使 DCache 无效化
   - 调用 `LCD_DrawBuffer()` 将帧数据刷新到 LCD

### 3.2 LCD ILI9341 驱动 (SPI1 + DMA)

- 通信方式: SPI1 (仅发送, 无 MISO) + DMA1_Stream0 加速
- 屏幕尺寸: 240×320
- 颜色格式: RGB565 (16bit)
- SPI 时钟: APB2/8 ≈ 17.2MHz
- DMA 通道: DMA1_Stream0, DMAMUX request = DMA_REQUEST_SPI1_TX (38)

**DMA 传输架构:**
- Ping-pong 双缓冲: `lcd_dma_buf[2][1024]` (每个 buffer 512 像素)
- 缓冲区位置: AXI SRAM 0x24040000 (DTCM 不可被 DMA 访问!)
- 传输前调用 `SCB_CleanDCache_by_Addr()` 刷新 DCache
- DMA 完成中断 → `HAL_SPI_TxCpltCallback()` → 清除 busy 标志
- `LCD_DMA_Wait()` 带超时保护，防止 DMA 异常时死循环

**批量传输流程 (LCD_DrawBuffer):**
1. 设置窗口 (阻塞 SPI 发送命令字节)
2. 切换 DC 为数据模式，保持 CS 低
3. 循环: CPU 在 buf[idx] 做字节序转换 → 等待上一次 DMA 完成 → 启动 DMA 发送 → 切换 idx
4. 等待最后一次 DMA 完成 → 释放 CS

**中断配置:**
- DMA1_Stream0_IRQHandler → `HAL_DMA_IRQHandler(&hdma_spi1_tx)`
- SPI1_IRQHandler → `HAL_SPI_IRQHandler(&hlcd_spi)` (错误处理)

### 3.3 I2C 软件模拟驱动

- 实现方式: GPIO bit-bang (非硬件 I2C)
- 引脚: PB8 (SCL) / PB9 (SDA), 开漏 + 内部上拉
- 速率: ~100kHz (通过 `I2C_DELAY_US = 5` 控制)
- 源文件: `Communication/I2C.c`

**API 接口:**
```c
void    I2C_Sim_Init(void);                                              // GPIO 初始化
uint8_t I2C_WriteData(uint8_t dev_addr, uint8_t reg_addr, uint8_t *data, uint16_t len);
uint8_t I2C_ReadData(uint8_t dev_addr, uint8_t reg_addr, uint8_t *buffer, uint16_t len);
```

> 注意: `dev_addr` 为 7 位地址，函数内部自动左移并添加 R/W 位。

### 3.4 MPU6050 六轴传感器驱动

- 通信接口: 软件 I2C (PB8/PB9)
- 设备地址: 0x68
- 采样率: 100Hz
- 陀螺仪量程: ±2000°/s
- 加速度计量程: ±16g
- 数字低通滤波: DLPF 带宽 5Hz
- 源文件: `Communication/MPU6050.c`

**初始化流程:**
1. 写 `PWR_MGMT_1 = 0x01` → 退出睡眠，选择 X 轴陀螺仪 PLL 作为时钟源
2. 写 `PWR_MGMT_2 = 0x00` → 使能所有轴
3. 写 `SMPLRT_DIV = 0x09` → 采样率分频 (100Hz)
4. 写 `CONFIG = 0x06` → DLPF 配置
5. 写 `GYRO_CONFIG = 0x18` → 陀螺仪 ±2000°/s
6. 写 `ACCEL_CONFIG = 0x18` → 加速度计 ±16g

**数据读取:**
- `MPU6050_GetData()` — 逐寄存器读取 (12 次 I2C 事务)
- `MPU6050_GetDataFast()` — 从 0x3B 连续读取 14 字节 (1 次 I2C 事务，推荐使用)

**寄存器映射 (连续读取 buffer[0..13]):**

| 偏移 | 内容 |
|------|------|
| [0:1] | AccX (高:低) |
| [2:3] | AccY |
| [4:5] | AccZ |
| [6:7] | Temperature (未使用) |
| [8:9] | GyroX |
| [10:11] | GyroY |
| [12:13] | GyroZ |

### 3.5 LVGL 集成

- 版本: v8.3.11
- 颜色深度: 16 bit (RGB565)
- 显示端口: `lv_port_disp.c` (位于 `LVGL/examples/porting/`)
- 刷屏回调: 调用 `LCD_SetWindows()` + `LCD_WriteDataBuffer()`

### 3.5.1 TIM7 LVGL 心跳定时器

- 源文件: `SYSTEM/TIM.c`, `SYSTEM/TIM.h`
- 定时器: TIM7 (基本定时器, 挂在 APB1 总线)
- 实现方式: 直接寄存器操作 (未启用 `HAL_TIM_MODULE_ENABLED`, 减少代码体积)
- 时钟来源: APB1 Timer Clock = 275MHz (APB1 分频≠1 时, Timer Clock = 2 × APB1)

**定时器参数:**

| 参数 | 值 | 说明 |
|------|-----|------|
| PSC | 274 | 275MHz / 275 = 1MHz 计数频率 |
| ARR | 999 | 1MHz / 1000 = 1kHz 中断频率 |
| 中断周期 | 1ms | 精确匹配 LVGL tick 需求 |
| NVIC 优先级 | (3, 0) | 低于 DCMI(0,x) 和 SPI(1,x) |

**API:**
```c
void TIM7_Init(void);       // 初始化并启动 TIM7, 在 lv_init() 之后调用
void TIM7_IRQHandler(void); // 中断服务函数, 调用 lv_tick_inc(1)
```

**使用方式:**
```c
#include "TIM.h"
// 在 lv_init() 之后调用
TIM7_Init();
```

> ⚠️ `TIM7_IRQHandler` 已在 `SYSTEM/TIM.c` 中定义，确保 `stm32h7xx_it.c` 中不要重复定义该函数。

### 3.7 AHT20 温湿度传感器

- 通信接口: 软件 I2C (PB8=SCL, PB9=SDA)
- 设备地址: 0x38 (7-bit)
- 供电: 3.3V
- 源文件: `Sensor/AHT20.c`, `Sensor/AHT20.h`

**协议特点:**
- AHT20 的 I2C 读取不使用标准寄存器地址方式
- 读状态/数据时直接发 [地址+R] 然后读 N 字节 (无需先写寄存器地址)
- 发命令时: [地址+W] + 命令字节 + 参数字节

**初始化流程:**
1. 上电等待 40ms
2. 发送初始化命令: `0xBE 0x08 0x00`
3. 等待 500ms
4. 检查状态字节的 CAL 位 (bit3)，未校准则软复位重试

**测量流程:**
1. 发送触发测量命令: `0xAC 0x33 0x00`
2. 等待 80ms (典型测量时间 75ms)
3. 轮询状态字节 bit7 (busy)，等待清零
4. 直接读取 7 字节: [状态][湿度H][湿度M][湿度L|温度H][温度M][温度L][CRC]
5. 解析 20 位湿度和 20 位温度原始数据

**数据转换:**
```c
// 湿度: humidity_x100 = raw_humi * 10000 / 2^20 (单位 0.01%)
// 温度: temp_x100 = raw_temp * 20000 / 2^20 - 5000 (单位 0.01°C)
// 注意: 必须用 uint64_t 中间运算防止 32 位溢出!
*humidity = (uint32_t)(((uint64_t)ct[0] * 10000) / 1048576);
*temperature = (int32_t)(((uint64_t)ct[1] * 20000) / 1048576 - 5000);
```

**踩坑记录:**
- `ct[0] * 10000UL` 在 32 位下会溢出 (最大值 1048575 × 10000 > UINT32_MAX)，导致湿度显示为负数
- 必须使用 `(uint64_t)ct[0] * 10000` 避免溢出

### 3.8 BH1750FVI 光照传感器

- 通信接口: 软件 I2C (PB8=SCL, PB9=SDA)，与 AHT20 共享总线
- 设备地址: 0x23 (ADDR 引脚接 GND)
- 供电: 2.4~3.6V (典型 3.0V)
- 量程: 1 ~ 65535 lx (H-Resolution Mode)
- 分辨率: 1 lx (H-Res), 0.5 lx (H-Res2), 4 lx (L-Res)
- 源文件: `Sensor/BH1750.c`, `Sensor/BH1750.h`

**I2C 地址选择:**

| ADDR 引脚 | 7-bit 地址 | 二进制 |
|-----------|-----------|--------|
| GND (≤0.3VCC) | 0x23 | 0100011 |
| VCC (≥0.7VCC) | 0x5C | 1011100 |

**指令集:**

| 指令 | Opcode | 说明 |
|------|--------|------|
| Power Down | 0x00 | 关机，无活动状态 |
| Power On | 0x01 | 等待测量命令 |
| Reset | 0x07 | 复位数据寄存器 (需先 Power On) |
| 连续 H-Res | 0x10 | 1lx 分辨率, 120ms |
| 连续 H-Res2 | 0x11 | 0.5lx 分辨率, 120ms |
| 连续 L-Res | 0x13 | 4lx 分辨率, 16ms |
| 单次 H-Res | 0x20 | 1lx, 120ms, 测完自动 Power Down |
| 单次 H-Res2 | 0x21 | 0.5lx, 120ms, 测完自动 Power Down |
| 单次 L-Res | 0x23 | 4lx, 16ms, 测完自动 Power Down |

**写命令格式:**
```
ST + [SlaveAddr + W] + Ack + [Opcode] + Ack + SP
```
> 重要: BH1750 不能在一次传输中接受多条命令，每条命令后必须插入 Stop!

**读数据格式:**
```
ST + [SlaveAddr + R] + Ack + [HighByte] + Ack + [LowByte] + NAck + SP
```

**光照计算公式:**
```c
// H-Resolution Mode:
lux = raw_data / 1.2 * (69 / MTreg)
// H-Resolution Mode2:
lux = raw_data / 1.2 * (69 / MTreg) / 2
```

**MTreg 灵敏度调节:**
- 默认值: 69 (0100_0101)
- 范围: 31 ~ 254
- 设置高位: 发送 `01000_MT[7:5]`
- 设置低位: 发送 `011_MT[4:0]`
- 灵敏度倍数 = MTreg / 69

**本项目使用方式:**
- 连续高分辨率模式 (0x10)，初始化时启动一次，之后每秒读取最新值
- 不调用 `BH1750_MeasureOnce()` (该函数内部用 `HAL_Delay`，在 FreeRTOS 中不安全)
- 使用 `BH1750_StartMeasurement()` + `BH1750_ReadLux()` 分离模式

### 3.9 传感器任务架构

**统一采集任务 (AHT20_Task):**
- AHT20 和 BH1750 在同一个 FreeRTOS 任务中顺序读取
- 共享 I2C 总线，单任务串行访问，无需互斥锁
- 采集周期: 1 秒
- 两个传感器独立初始化，任一个不在线不影响另一个工作

**数据流:**
```
Sensor_Task (1s 周期, 已替代旧的 AHT20_Task)
  ├── 读取 AHT20 温湿度
  ├── 读取 BH1750 光照
  ├── 读取 MAX9814 麦克风音量 (RMS → 0~100%)
  ├── 打包为 SensorData_t
  └── xQueueOverwrite(xSensorDataQueue)  → DisplayTask 消费 (LVGL 显示)
```

**数据结构:**
```c
typedef struct {
    int32_t  temperature;     // 温度, 单位 0.01°C
    uint32_t humidity;        // 湿度, 单位 0.01%
    uint32_t lux_x100;        // 光照, 单位 0.01 lx
    uint16_t soundIntensity;  // 麦克风音量 0~100 (百分比)
    uint32_t timestamp;       // FreeRTOS tick 时间戳
} SensorData_t;
```

**队列设计:**
- `xSensorDataQueue`: 长度 1，使用 `xQueueOverwrite` 始终保持最新值

**显示任务 (DisplayTask, 原 Weather_Task 已重命名):**
- 使用 `xQueuePeek` 读取最新数据 (不移除)
- 时间戳去重: 相同数据不重复刷新 UI
- 数据过期检测: 超过 3 秒未更新显示 "---" 离线状态
- LVGL 显示模式根据 `g_display_mode` 自动切换 (见 §3.12)
- 横屏 320×240 全屏天气仪表板, 左列网络天气 Hero 卡, 右列本地室内卡

**LVGL 显示布局 (天气模式, 横屏 320×240):**

### 3.10 蓝牙通信 (W02 / PB-03F) + 从机数据接收 (USART2)

- 模块: 安信可 PB-03F (或 W02) BLE 蓝牙透传模块
- 芯片: PHY6252 (博通集成)
- 协议: Bluetooth 5.2 BLE
- 通信方式: UART 透传
- 链路: ESP32 (从机, 采集 MPU6050 + 压力传感器) → BLE 广播 → 主机端 W02/PB-03F → USART2

**说明:**
- MPU6050 不在本机 I2C 总线上，而是在从机端 (ESP32) 采集后通过 BLE 透传到主机
- 主机 USART2 接收从机发来的 IMU + 压力 JSON 数据
- 因此 PB8/PB9 I2C 总线上只有 AHT20 和 BH1750，不存在与 MPU6050 的总线冲突

**PB-03F 关键参数:**

| 参数 | 值 |
|------|-----|
| 蓝牙版本 | Bluetooth 5.2 |
| 发射功率 | 10 dBm |
| 接收灵敏度 | -93 dBm |
| TX 电流 | 11.5 mA |
| RX 电流 | 9.4 mA |
| 支持速率 | 125K / 500K / 1M / 2Mbps |
| 默认工作模式 | UART 透传 |

**主机 USART2 接收链路:**

- 引脚: PA2 = TX, PA3 = RX, 115200 baud, 8N1
- DMA: DMA1_Stream3 (`DMA_REQUEST_USART2_RX`), 循环模式
- DMA 缓冲: AXI SRAM `0x24040D00` (256B, 32 字节对齐)
- 中断: USART2 IRQ (6, 1) + DMA1_Stream3 IRQ (6, 0), IDLE 检测帧结束
- 分层架构:
  - **驱动层** `Communication/UART_ESP32.{c,h}`: 字节流分帧 (USART2 + DMA + IDLE)
  - **业务层** `Task/SlaveRxTask.{c,h}`: cJSON 解析、长度 1 队列、`Slave_GetLatest/WaitNew/GetStats` API

**JSON 报文格式 (来自 ESP32):**
```json
{"ts":1678901234567,
 "ax":0.12,"ay":-0.05,"az":9.81,
 "gx":0.01,"gy":-0.02,"gz":0.03,
 "press":101.32}
```
> ESP32 端 sprintf 拼接可能带尾随逗号 `,}`, 业务层 `sanitize_trailing_comma` 已做容错。

**数据结构 (`SlaveData_t`):**
```c
typedef struct {
    uint64_t ts;            // ESP32 端毫秒时间戳 (超过 uint32 范围)
    float    ax, ay, az;    // 加速度, g
    float    gx, gy, gz;    // 角速度, dps
    float    press;         // 气压, kPa
    uint32_t local_tick;    // 本机接收时刻 FreeRTOS tick
    uint32_t seq;           // 累计帧序号, 便于丢帧统计
} SlaveData_t;
```

**接收流程:**
```
ESP32 + MPU6050 + 压力 → BLE 广播 → W02 / PB-03F 透传 → USART2 RX (PA3)
   ↓ DMA1_Stream3 → AXI SRAM 0x24040D00 (256B)
IDLE 中断 (UART_ESP32.c::USART2_IRQ_Handler)
   ↓ SCB_InvalidateDCache → 拷贝到帧缓冲 → vTaskNotifyGiveFromISR
Slave_RxTask (Task/SlaveRxTask.c) 唤醒
   ├── UART_ESP32_TakeFrame(buf, max_len, &out_len)
   ├── sanitize_trailing_comma + cJSON 解析 → SlaveData_t
   ├── printf 调试到 USART1
   └── xQueueOverwrite(s_queue) (内部, 长度 1)
       ↓
   消费方调用 Slave_GetLatest() / Slave_WaitNew() 获取最新一帧
```

### 3.11 ESP-01 WiFi 天气数据 (USART3)

- 模块: ESP-01 (ESP8266 WiFi 模块)
- 通信接口: USART3 (PD8=TX, PB11=RX, 115200 baud)
- 协议: MQTT 透传 JSON 格式天气数据
- 接收方式: DMA 循环模式 + 空闲中断 (IDLE) 检测帧结束
- 分层架构 (重构后):
  - **驱动层** `Communication/UART_ESP01.{c,h}`: USART3 + DMA1_Stream1 + IDLE 分帧
  - **业务层** `Task/WeatherTask.{c,h}`: cJSON 解析、长度 1 队列、`Weather_GetLatest/WaitNew/GetStats` 消费 API
  - 驱动只负责"字节流分帧", 不参与协议解析; 业务侧通过 `UART_ESP01_BindRxTask + UART_ESP01_TakeFrame` 与驱动对话

**JSON 数据格式:**
```json
{"city":"Guangzhou","temp":29.0,"humi":76,"feel":33,"cond":"Patchy rain nearby","wind":4.0,"wdir":"E","pres":1009,"time":"00:05"}
```

**字段说明:**

| 字段 | 类型 | 说明 |
|------|------|------|
| city | string | 城市名 |
| temp | float | 温度 (°C) |
| humi | int | 湿度 (%) |
| feel | float | 体感温度 (°C) |
| cond | string | 天气状况描述 |
| wind | float | 风速 (m/s) |
| wdir | string | 风向 (E/W/S/N等) |
| pres | int | 气压 (hPa) |
| time | string | 时间 |

**接收流程:**
```
ESP-01 → USART3 RX (PB11) → DMA1_Stream1 → AXI SRAM 0x24040800 (256B)
   ↓ IDLE 中断 (UART_ESP01.c::USART3_IRQ_Handler)
SCB_InvalidateDCache → 拷贝到帧缓冲 → vTaskNotifyGiveFromISR
   ↓
Weather_RxTask (Task/WeatherTask.c) 唤醒
   ├── UART_ESP01_TakeFrame(buf, max_len, &out_len)
   ├── cJSON 解析 → WeatherInfo_t
   ├── printf 调试到 USART1 (走非阻塞 fputc)
   └── xQueueOverwrite(s_queue)
       ↓
   DisplayTask (Task/Display.c) 通过 Weather_GetLatest() 拉取, 更新 LVGL UI (city, temp, humi 等)
```

**踩坑记录:**

1. **DMA 缓冲区位置**: `rx_dma_buf` 必须放在 AXI SRAM 而非 DTCM, 使用
   `__attribute__((section(".ARM.__at_0x24040800"), aligned(32)))` 强制定位。
   紧跟 LCD DMA buffer (0x24040000~0x240407FF) 之后。
2. **DCache 一致性**: DMA 写入 AXI SRAM 后, CPU 读取前必须调用
   `SCB_InvalidateDCache_by_Addr()` 否则读到旧数据。
3. **中断优先级**: USART3 / DMA1_Stream1 优先级设为 6, 高于 FreeRTOS
   `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY (5)`, 才能调用 FromISR API。
4. **GPIO 复用**: PD8=TX 需要使能 GPIOD 时钟; PB11=RX 使能 GPIOB 时钟。
5. **IDLE 中断**: HAL_UART_Receive_DMA() 启动循环 DMA 后, 必须手动
   `__HAL_UART_ENABLE_IT(&huart3, UART_IT_IDLE)` 启用空闲检测。
6. **每帧重启 DMA**: IDLE 触发后 `HAL_UART_DMAStop` 再 `HAL_UART_Receive_DMA` 重启,
   重置 DMA 计数器以接收下一帧。

**WeatherInfo_t 结构体:**
```c
typedef struct {
    char     city[20];      // 城市名
    int16_t  temp_x10;     // 温度, 单位 0.1°C (290 = 29.0°C)
    uint8_t  humi;         // 湿度, %
    int16_t  feel_x10;    // 体感温度, 0.1°C
    char     cond[32];     // 天气状况
    uint16_t wind_x10;    // 风速, 0.1 m/s
    char     wdir[4];      // 风向
    uint16_t pres;         // 气压, hPa
    char     time[8];      // 时间字符串
    uint32_t timestamp;   // 接收时刻 tick
} WeatherInfo_t;
```

**cJSON 适配 FreeRTOS:**
- 在 main.c 中通过 `cJSON_InitHooks()` 注册 `pvPortMalloc` / `vPortFree`
- 解析时分配的内存来自 FreeRTOS 堆 (heap_4)
- `cJSON_Delete()` 自动释放

### 3.12 显示模式切换 + USART1 非阻塞 printf

> 本节描述 V1.3 引入的运行时显示切换机制和 USART1 的非阻塞输出实现。

#### 3.12.1 模式定义

| 模式 | 命令字符 | LCD 方向 | 显示内容 |
|------|---------|----------|---------|
| `DISP_MODE_CAMERA` | `'C'` / `'c'` | 1 (横屏 90°) | 240×240 摄像头画面居中, 左右各 40px 留白 |
| `DISP_MODE_WEATHER` | `'W'` / `'w'` | 3 (横屏 270°) | 320×240 全屏天气仪表板 |

> 两种模式之间相对旋转 180°, 物理屏幕在横屏放置时无论哪个方向朝上都正向显示一种内容。

#### 3.12.2 数据结构与 API

```c
/* Task/display_mode.h */
typedef enum {
    DISP_MODE_CAMERA = 0,
    DISP_MODE_WEATHER = 1,
} DisplayMode_t;

extern volatile DisplayMode_t g_display_mode;

void DisplayMode_Request(DisplayMode_t mode);                 /* ISR 安全 */
int  DisplayMode_TakePending(DisplayMode_t *out_mode);        /* 仅 LVGL 任务调用 */
void DisplayMode_StartUart1Rx(void);                          /* main.c 中 USART1_Init() 后调用 */
void USART1_IRQ_Handler(void);                                /* 由 stm32h7xx_it.c 转发 */
```

#### 3.12.3 切换流程

```
USART1 RX ('C' / 'W')
  └→ USART1_IRQHandler → USART1_IRQ_Handler
       └→ process_rx_byte → DisplayMode_Request(mode)
            └→ 设置 s_pending_flag/s_pending_mode (volatile)

DisplayTask 主循环 (100ms, 原 Weather_Task)
  └→ DisplayMode_TakePending(&pending)
       └→ Weather_ApplyMode(pending)
            ├→ LCD_direction(1 或 3)
            ├→ lv_obj_add/clear_flag(canvas, HIDDEN)
            ├→ lv_obj_add/clear_flag(weather_panel, HIDDEN)
            └→ lv_obj_invalidate(lv_scr_act())  ← 强制整屏重绘
```

> ISR 只设置请求标志, **不直接调用 LVGL API**, 保证 LVGL 单线程访问规则。

#### 3.12.4 Camera_task 配合

- DCMI / DMA 始终连续运行 (避免重启硬件代价)
- 通过 `g_cam_canvas` 全局对象由 DisplayTask 控制 hidden
- Camera_task 内只在 `g_display_mode == DISP_MODE_CAMERA` 时才 `memcpy` + `lv_obj_invalidate(canvas)`,
  天气模式下减少 LVGL 工作量

#### 3.12.5 USART1 非阻塞 printf (TX)

**为什么需要重写 fputc:**
- 原始 fputc 调用 `HAL_UART_Transmit(huart1, ..., 100ms)`, 是阻塞调用
- 多任务并发 printf 时 HAL 的 `gState` 互斥保护不可靠 (任务被高优先级抢占可能导致 gState 卡在 BUSY_TX)
- 表现为: 发送某个字符 / 收到 MQTT 消息后串口完全卡死, printf 不再输出

**新实现 (`Task/display_mode.c` 中的 fputc):**
- 2KB 静态环形缓冲 `tx_ring[2048]` (大小为 2 的幂, 用 `&` 取模)
- 任务调 printf → fputc 把字节写入 `tx_ring`, 立即返回 (微秒级)
- 写入时使能 USART1 的 TXE 中断 (`USART_CR1_TXEIE_TXFNFIE`)
- 短临界区 (`__disable_irq` + 改 head + `__enable_irq`) 保护多任务并发
- 自动 `\n` → `\r\n` 转换 (省得用户每条 printf 都写 `\r\n`)
- 缓冲满时丢弃字节 (不阻塞调用者), 不会导致死锁

**TX 中断 ISR 路径:**
```
USART1 TXE 触发
  └→ USART1_IRQ_Handler
       ├→ if (RXNE) 处理命令字符 (RX 路径)
       ├→ if (TXE && tx_head != tx_tail)
       │     └→ TDR = tx_ring[tx_tail]; tx_tail++
       └→ if (tx_head == tx_tail) 关闭 TXE 中断 (避免空转)
```

**RX 路径:**
- 直接寄存器级使能 `USART_CR1_RE` + `USART_CR1_RXNEIE_RXFNEIE`
- **不**调用 `HAL_UART_Receive_IT` (该函数会和 fputc 的 huart1 状态机冲突)
- 自定义 `USART1_IRQHandler` 直接读 `USART1->RDR`, 处理 `'C'`/`'W'` 命令
- 错误标志 (ORE/FE/NE/PE) 在 ISR 中清掉, 防止反复触发

**踩坑记录:**
1. **HAL 状态机冲突**: 同时用 `HAL_UART_Receive_IT` (RX) + 阻塞 `HAL_UART_Transmit` (TX) 在多任务下不可靠, 必须绕开 HAL 状态机直接操作寄存器。
2. **USART_CR1 位**: STM32H7 用 `USART_CR1_RXNEIE_RXFNEIE` (FIFO 架构), 不是 F1/F4 的 `USART_CR1_RXNEIE`。
3. **PA10 浮空检测**: 初版调试时遇到 PA10 一直为 0, 通过临时配置 INPUT+NOPULL/PULLUP/PULLDOWN 三种模式读电平, 准确判断接线问题。该自测代码已移除。
4. **fputc 多重定义**: 旧 fputc 在 `Drivers/User/Src/usart.c` 里, 新的在 `Task/display_mode.c`。旧的已用 `#if 0` 注释禁用, 避免链接器多重定义错误。
5. **环形缓冲大小**: 必须是 2 的幂 (用 `&` 替代 `%` 提速), 当前 2048 字节足以应付 ESP-01 触发的 `print_weather_info` (9 行 printf, 约 200 字节) 突发输出。

方案 B — 将 `lv_conf.h` 移动到 `LVGL` 文件夹的上一级目录 (即项目根目录)。

### 问题 5: DCache 一致性

**现象:** (潜在问题) DMA 写入帧缓冲后 CPU 读取到旧数据。

**定位:** DMA 直接写入 AXI SRAM，CPU 通过 DCache 读取可能命中缓存中的旧数据。

**原因:** DMA 传输绕过 CPU Cache，导致 Cache 与内存不一致。

**解决:** 每帧 DMA 完成后调用:

```c
SCB_InvalidateDCache_by_Addr((uint32_t *)Camera_Buffer, Display_Width * Display_Height * 2);
```

**验证:** 已在 `Camera_task` 中实现，图像显示正常无花屏。

### 问题 6: SPI DMA 导致 DCMI 完全不工作 ⭐ 重要

**现象:** 改用 SPI DMA 传输后，DCMI_FrameState 始终为 0，Camera_Buffer 无任何数据，屏幕无显示。

**定位:** Keil Debug 观察变量，发现 Camera_task 卡死在 `LCD_Init()` → `LCD_Clear()` → `LCD_DMA_Wait()` 中，`DCMI_OV2640_Init()` 从未被执行。

**原因:** STM32H7 的 **DTCM RAM (0x20000000) 不能被 DMA1/DMA2 访问**！Scatter file 将所有 RW/ZI 数据放在 DTCM，导致 `lcd_dma_buf` 在 DTCM 中。DMA1 从 DTCM 读数据时静默失败，`HAL_SPI_TxCpltCallback` 永远不触发，`lcd_dma_busy` 卡在 1，`LCD_DMA_Wait()` 死循环。

**解决:**
1. DMA 缓冲区使用 `__attribute__((section(".ARM.__at_0x24040000")))` 强制放到 AXI SRAM
2. Scatter file 添加 AXI SRAM 执行区域:
```
RW_AXIRAM 0x24040000 0x00010000 {
    *(.ARM.__at_0x24040000)
}
```
3. `LCD_DMA_Wait()` 增加超时保护
4. 添加 `HAL_SPI_ErrorCallback` 防止错误时死锁
5. 添加 SPI1_IRQHandler 处理 HAL DMA 使能的 SPI 错误中断

**验证:** 修改后摄像头采集和 LCD DMA 刷屏均正常工作。

**教训:** 在 STM32H7 上使用 DMA 时，必须确认缓冲区所在的内存区域是否可被 DMA 访问。DTCM 仅 CPU 可访问，DMA 缓冲区必须放在 AXI SRAM、SRAM1~3 等 DMA 可达区域。

## 十一、重构记录 (V1.4 — UART 驱动业务分层 + 任务命名规范)

### 11.1 背景

V1.3 之前的代码存在两个分层问题:

1. **UART 接收逻辑与业务解析混杂**: `Sensor/UART.c` 同时承担 USART3 底层驱动 (DMA + IDLE) 和天气 JSON 解析、队列发布、调试打印。新增 ESP32 BLE 链路时，再写一份类似但又不同的代码会带来重复。
2. **任务文件名与职责不匹配**: `Task/WeatherTask.c` 是 LVGL 仪表板渲染任务，与"天气数据接收"在概念上是两件事; `LCD/touch_test.c` 名字像测试代码，实际上是触摸驱动主任务。

### 11.2 重构策略

**UART 类业务统一拆成驱动层 + 业务层:**

| 链路 | 驱动层 (`Communication/`) | 业务层 (`Task/`) |
|------|--------------------------|------------------|
| ESP-01 WiFi (USART3) | `UART_ESP01.{c,h}` 分帧 | `WeatherTask.{c,h}` 解析 + 队列 + 消费 API |
| ESP32 BLE (USART2) | `UART_ESP32.{c,h}` 分帧 | `SlaveRxTask.{c,h}` 解析 + 队列 + 消费 API |

驱动层 API 模板:
```c
void  USARTx_Init(void);
void  USARTx_IRQ_Handler(void);
void  UART_xxx_BindRxTask(TaskHandle_t handle);
int   UART_xxx_TakeFrame(uint8_t *buf, uint16_t max_len, uint16_t *out_len);
void  UART_xxx_GetStats(uint32_t *recv, uint32_t *drop);
```

业务层消费 API 风格:
```c
int  Xxx_GetLatest(XxxData_t *out);                       /* 非阻塞 peek */
int  Xxx_WaitNew(XxxData_t *out, TickType_t timeout);    /* 阻塞 receive */
void Xxx_GetStats(uint32_t *recv, uint32_t *drop, uint32_t *parse_err);
```

新增 UART 链路按此模板复制即可。

**LVGL 显示任务从 WeatherTask 拆分独立**: 渲染逻辑迁移到 `Task/Display.c::DisplayTask`, 数据通过消费 API 拉取，不再直接访问全局队列。

**触摸任务按统一风格命名**: `LCD/touch_test.{c,h}` → `Task/TouchTask.{c,h}`, `TouchTest_*` API 重命名为 `Touch_*`。
